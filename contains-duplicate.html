#217. Contains Duplicate

Easy Arrays Hashing
⏱ 4 min read 📐 3 approaches 🏢 Amazon · Apple · Google ✅ ~61% acceptance
Try on LeetCode ↗

Problem Statement

Given an integer array nums, return true if any value appears at least twice in the array, and false if every element is distinct.

Example 1:
Input: nums = [1,2,3,1]
Output: true — the value 1 appears at index 0 and index 3.

Example 2:
Input: nums = [1,2,3,4]
Output: false — all values are distinct.

Constraints: 1 ≤ nums.length ≤ 10⁵ · −10⁹ ≤ nums[i] ≤ 10⁹

Intuition — Building the Mental Model

Before writing a single line of code, ask yourself: what does it mean for a duplicate to exist?

A duplicate simply means a value you've seen somewhere earlier in the array shows up again. That's all. You don't care about positions, counts, or order — just has this value appeared before?

The Karpathy Reframe

Don't think "find a duplicate pair." Think "maintain a memory of what you've seen. The moment you encounter something in memory — stop, return true." This single reframe gives you the optimal algorithm immediately.

A set is exactly "memory of things seen" — it stores unique values, ignores duplicates, and answers "have I seen X before?" in O(1). You iterate once, you're done.

This is the first of many problems where a set/map transforms O(n²) to O(n). Master this pattern here — you'll use it in Two Sum, Longest Consecutive Sequence, Group Anagrams, and beyond.

Approach 1 — Brute Force

For every element nums[i], scan every later element nums[j] and check if they're equal. O(n²) — never mention this as your solution in an interview unless asked.

Python 
def containsDuplicate(nums):
    n = len(nums)
    for i in range(n):
        for j in range(i + 1, n):
            if nums[i] == nums[j]:
                return True
    return False

Complexity: Time O(n²) · Space O(1). Too slow for n = 10⁵.

Approach 2 — Sort then Compare Neighbors

After sorting, any duplicate values must be adjacent. One scan after sorting is enough.

Why this works

Sorting groups identical values together. If any value repeats, it and its copy become neighbors. One forward scan catches any identical adjacent pair.

Python 
def containsDuplicate(nums):
    nums.sort()          # O(n log n) — modifies original
    for i in range(1, len(nums)):
        if nums[i] == nums[i - 1]:
            return True
    return False

Complexity: Time O(n log n) · Space O(1) with in-place sort, but note it mutates the input array. Mention this tradeoff in interviews.

Approach 3 — HashSet (One Pass) ★ Optimal

Maintain a set of values seen so far. For each element: if it's in the set, we found a duplicate — return True. Otherwise, add it to the set and continue.

Python 
def containsDuplicate(nums):
    seen = set()
    for n in nums:
        if n in seen:
            return True
        seen.add(n)
    return False

# One-liner (same logic, less readable):
def containsDuplicate(nums):
    return len(nums) != len(set(nums))
Java 
public boolean containsDuplicate(int[] nums) {
    Set<Integer> seen = new HashSet<>();
    for (int n : nums) {
        if (!seen.add(n)) // add() returns false if already present
            return true;
    }
    return false;
}
C++ 
bool containsDuplicate(vector<int>& nums) {
    unordered_set<int> seen;
    for (int n : nums) {
        if (seen.count(n)) return true;
        seen.insert(n);
    }
    return false;
}

Complexity: Time O(n) · Space O(n). One pass through the array. This is the answer to give in interviews.

Edge Cases to Test

1. Single element: [1] → False. No pair can exist with one element.

2. All same: [7,7,7,7] → True. Caught at step 2.

3. Duplicate at the very end: [1,2,3,4,5,1] → True. The algorithm still finds it — it scans all the way through.

4. Negative numbers: [-1, -2, -3, -1] → True. HashSets handle negative integer keys perfectly. Don't forget to test these.

Interview Follow-Up Questions

  • ❓
    What if you couldn't use extra space? Use the sort approach. O(n log n) time, O(1) extra space (in-place sort). Mention that it mutates the array.
  • ❓
    What if you need to return the duplicate value itself (not just true/false)? With the set approach, when you detect n in seen, just return n instead of True.
  • ❓
    Contains Duplicate II (#219) — what if duplicates must be within k positions of each other? Use a sliding window set of size k. Add new elements, remove elements more than k steps old. O(n) time.
  • ❓
    Contains Duplicate III (#220) — what if values must be within t of each other? This is much harder — requires a sorted data structure (TreeSet in Java) or bucket sort trick. O(n log k) time.

The Pattern — Where Else Does This Apply?

Pattern: "Have I Seen This Before?"

Any time you need O(1) membership queries — "does X exist in my collection of seen items?" — reach for a HashSet. This pattern powers: Two Sum (hash map for complement lookup), Longest Consecutive Sequence (set membership to skip non-starts), Valid Anagram (frequency counting), and First Missing Positive. The common thread: replace O(n) scan with O(1) lookup by trading space for time.
